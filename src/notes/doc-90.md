This video describes how Vercel built its internal text-Q&A-to-SQL agent (D0) step by step, eventually evolving it into a new agent framework — "Next.js for agents."

The speaker starts by describing the series of AI infrastructure pieces Vercel has built: the AI SDK (switch models and providers with one line of code), the AI Gateway (unified routing to different model providers), Sandbox (a safe execution environment for untrusted code), and Workflows (long-running, resumable workflows). These were all built to cut down on repeated engineering while they built their own AI products and agents.

The real story starts from a pain point: the internal data team kept getting interrupted by other departments (sales, marketing, PM, design, engineering) asking for help with one-off metrics and reports. The team felt this "ask a question → write SQL → query Snowflake → report back" workflow was a great fit for agent automation, so they set out to build an internal text-to-SQL agent.

Their first version was blunt: paste the entire Snowflake semantic layer (about 300 entities) straight into an LLM prompt and ask the model to produce SQL. The result wasn't good — the context was too large and too noisy. After they narrowed and compressed the search scope, results improved, giving them the confidence to keep evolving it.

Next, they broke down "the thought process a data scientist goes through when answering a question" into multiple sub-agents: understand the question, explore the semantic layer, design the query and joins, generate SQL, validate, execute, and report back. Each step was its own sub-agent, with the previous step's output feeding into the next. This improved quality somewhat, but debugging across multiple stages was extremely painful.

As model context windows grew larger, they consolidated the pipeline back into a single agent, but with "multiple stages" inside one execution loop: in the early part of the run the agent has many tools available (exploration, reasoning, query design); once SQL has been produced, the extra tools are retracted, leaving only the essential capabilities needed to "execute against Snowflake and report back." You can think of this as progressively narrowing the available toolset as execution advances.

Even so, they still weren't comfortable rolling it out to the whole company. Later, once Claude Code (paired with Opus 4.5) appeared, they tried using Claude Code directly as the core of the data-science agent, and evaluated it with their existing eval set (real questions drawn from existing data dashboards). Compared with all their previous multi-agent, multi-stage versions, Claude Code could handle almost every question in a single pass, and eval scores jumped substantially.

After analysis, they found the key wasn't some magic prompt — it was the file system: Claude Code treats information like schemas and semantic layers as files, and reasons step by step by reading and writing files. So they took this design and completely rebuilt their own D0: instead of maintaining a pile of fine-grained tools (join-pattern analysis, loading the semantic layer, exploration, checking Snowflake relationships, and so on), they dump all the needed information into files inside a sandbox, and give the agent only a small handful of basic capabilities: read file, write file, plotting, and running bash. This "minimal tools + file system" version performed very well — essentially hitting product-market fit — and they could confidently roll it out to more Vercel employees.

After that, D0 started handling about 2,000 queries a day internally (the video description says about 1,200, but the talk itself mentions it has since grown to 2,000). As usage grew, they observed clear categories emerging in question types: customer questions tend to query Salesforce, trend and time-series questions rely on particular aggregation patterns, and so on. So they began "distilling" these recurring patterns into reusable skills, preloaded into D0, so the agent doesn't have to re-reason "what type of question is this" every single time — it can just apply an existing skill directly. They even automated this process: scanning company-wide questions every day, extracting themes, and continually adding new skills. They've now accumulated more than 40 skills, covering everything from customer-facing to product-facing analytical needs.

While D0 was evolving, many teams inside Vercel also wanted to "fork D0 and build their own agent." This exposed another problem: even with a lot of powerful building blocks available (AI SDK, Gateway, Sandbox, Workflows, etc.), "correctly wiring them together" was still painful for users, who often used them incorrectly or ended up with strange architectures. This reminded the team of the role Next.js plays: wrapping a pile of web technologies into one opinionated framework so everyone uses them the recommended way.

So they asked themselves: could they build something like a Next.js, but for agents? Something that lets you build the correct agent architecture with minimal boilerplate and minimal configuration, while getting durable compute, sandbox security, multi-tool and sub-agent support, plus a standardized chat interface and multi-channel output for free.

They showed an early prototype of this new framework: an agent is just a directory containing agent.ts, system.md, and subdirectories like skills, tools, sandboxes, channels, and connections. Everything is automatically assembled and compiled via file-system-based routing — you don't need to define endpoints yourself or manually wire up Slack, web chat, etc. Under the hood, the framework uniformly uses workflows to provide observability and durability; on top of that, you can use what they call the chat SDK, letting a single piece of code ship to Discord, Slack, the web, and — in the future — chat platforms like Telegram and WhatsApp.

Finally, he mentions that D0 has now been "rebased" onto this new agent framework: what used to be twenty- or thirty-thousand-plus lines of scattered code has been condensed down into a pile of markdown documents, a small amount of TypeScript scripts/tools, and some channel configuration defining things like how to respond in Slack and how to render tables. The whole journey illustrates a typical path: start from a simple prompt, gradually add complexity (multiple agents, multiple tools), then extract the abstraction that actually works (a file system + a handful of primitive capabilities + skills), and finally package it all into an opinionated framework so the whole company — and outside users — can build agents "the right way."
