The Agent Attack Surface: Why AI Is Breaking Software Security As We Know It

Feross, Socket

✕

This video is about how AI agents are fundamentally changing — even breaking — the software security model we're familiar with, especially around the software supply chain and vulnerability management.

At the start of the talk, Feross points out that the number of software supply-chain attacks has clearly "accelerated dramatically" since GPT-4. Modern applications often have more than 90% of their code coming from open-source dependencies (NPM, etc.), and on average each package pulls in dozens of "transitive dependencies" — code you never directly use but that gets dragged in anyway. Human developers already almost never looked at the source of these dependencies; what's worse now is that AI agents choose, install, and run packages themselves, often running with developer-level privileges with no human review at all.

He uses several cases to show just how dangerous this new attack surface is. Trustfall's research, for example, shows that clicking "I trust this folder" just once in an MCP-enabled AI coding tool can effectively hand a remote server full OS-level access. Similarly, malicious packages on NPM have exploded, with attackers doing what amounts to "LLM SEO (LLMO)": deliberately writing READMEs that are highly persuasive to models, even test-running them against GPT/Claude first to see which description gets picked most easily. The result is that AI ends up favoring packages that "look perfect" but actually hide malicious code, or have a trojan buried in the dependency tree.

He cites several real attacks:
a malicious NPM package that directly steals Linux's ‎⁠passwd⁠ file and sends it back to the attacker via curl; a more advanced version that uses obfuscation, fetches an executable from a Discord CDN, and then uses a child process to plant a backdoor on the system. There's also the North Korean APT campaign PromptMink: a top-level package that looks clean, with the malicious payload buried down in the dependency chain, and a README written even more carefully than its competitors' — the kind of thing that looks a bit suspicious to a human but reads as "the perfect recommendation" to an LLM, and it genuinely turned up in a Solana project written with Claude's help.

Feross then introduces the defenses Socket has built, such as "Socket Firewall": it puts a proxy layer in front of commands like ‎⁠npm install⁠, so every install request goes through static analysis and AI review, and anything flagged as a typosquat, a suspicious install script, or a release from a brand-new account gets blocked outright — effectively adding a security gatekeeper for package selection, for both humans and agents.

Midway through, he shifts focus from packages themselves to "the infrastructure underpinning agents": things like MCP, skills, and IDE extensions. Many teams rushed to integrate MCP without thinking through trust boundaries — for instance, one project added MCP endpoints but forgot to add authentication to one of them, and as a result thousands of instances were genuinely exposed on the open internet, where anyone could use an MCP message to rewrite Nginx configuration and redirect traffic to an attacker. Worse still, current MCP clients mix the output of multiple servers into the same LLM context with no isolation at all, so an apparently harmless "weather tool" can have a prompt injection stuffed into its returned content, manipulating permissions that were only ever meant for a database tool.

Skills/agent instruction marketplaces are another "new NPM." Sites like skills.sh host tens of thousands of skill files, most of which are just markdown — but that natural language is really "executable malicious logic." He shows a few examples: one skill instructs the agent, in its description, to silently run curl with no output shown; another appears to look up prediction-market data, but the Python script bundled with it slips in a line that pipes an attacker's shell script through curl straight into local execution. If a security tool only scans the markdown and doesn't look at the whole file set, it simply won't catch this. Compared to code libraries, skills typically run in the user's full environment without necessarily having container isolation, which makes the potential damage even worse.

He then turns to a key trend: AI isn't just expanding the attack surface, it's also "super-accelerating vulnerability discovery." Systems like Mythos can find bugs in mature projects such as FreeBSD that went unnoticed for years, and there are even cases confirmed by Google TAG showing that attackers are already using AI to help build real zero-days. The result is that the time "from public disclosure to exploitation" has shrunk from an average of over two years down to roughly 10 hours today. The traditional workflow of "file a ticket, schedule it, patch it by hand" simply can't keep up.

At the same time, the number of CVEs is exploding, and NIST's National Vulnerability Database (NVD) recently even announced that it would stop doing detailed enrichment for most CVEs — essentially giving up on fully curating and assessing the bulk of vulnerabilities. This means that security workflows relying on traditional scanning plus NVD data are suffering a systemic breakdown at precisely the moment the AI era needs them most.

He criticizes a core problem with traditional vulnerability scanners: the vast majority (he cites over 80%) of the vulnerabilities they report are actually "unreachable" — the flaw exists in some dependency's function, but the application never actually calls that function. Security teams and developers are nonetheless forced to spend enormous amounts of time triaging and upgrading versions, doing "busywork" that barely improves actual risk. In a world where AI keeps amplifying the sheer volume of vulnerabilities, this approach is completely unsustainable.

The core of his proposed solution is to promote "reachability analysis." The idea: use static analysis to build a call graph, and determine whether there's an actual path from your application code that reaches the "vulnerable function." Only if such a path exists does that CVE actually "need fixing." This typically filters out 80-90% of the vulnerabilities a tool would otherwise report, leaving only the 10-20% that are genuinely exploitable — letting teams prioritize the problems attackers can actually reach, instead of chasing noise.

To avoid the situation where "upgrading just creates a new attack surface," he argues that the future should lean more on "targeted, local patches" rather than blindly upgrading to a new version. Since most security fixes actually change only a line or two of code, automatically backporting these fixes to the version you're currently running can eliminate the vulnerability without taking on the large behavioral changes and new supply-chain risk that come with a full version upgrade. AI is well-suited to drive exactly this kind of precise patching and automatic modification of dependency code.

Finally, he sums up where things stand:
in the short term, AI will confront us with more malicious packages, more automated attacks, more vulnerabilities, and shorter response times, and the traditional compliance mindset of "every CVE must be fixed" has to shift to "accept that some vulnerabilities will always exist, and focus on the small subset that's reachable and exploitable." But in the long run, he's optimistic. Software vulnerabilities, while numerous, aren't infinite, and fixing each one theoretically removes one usable attack surface — and AI doesn't just help attackers write exploits, it can also help defenders review PRs, spot code smells, run static analysis and reachability checks, and generate patches. As these defensive agents get woven deeper into the development process, new bugs will become rarer and old bugs will get steadily cleared out; he believes we have a real shot at heading toward a "software world safer than it's ever been" — we're just in a painful transition period right now.
