AI Needs a New Kind of OLTP: Lakebase & Serverless Postgres in the Agent Era

Databricks (Lakebase/Neon)

✕

This talk explains why traditional OLTP databases (especially the typical way Postgres is deployed) are no longer a good fit for the new wave of applications driven by AI agents, and how Databricks' Lakebase/Neon solves this by splitting Postgres into a cloud-native architecture with storage and compute separated.

The speaker first explains why Lakebase/Neon is so popular among "coding AI agents": platforms like the Replit agent or Vercel v0 automatically create a database when they "auto-generate an app" for a user, and today over 80% of such databases are created automatically by agents rather than by a human manually. This produces a traffic pattern completely unlike traditional SaaS: huge numbers of short-lived databases, extremely bursty, environments spun up and torn down very quickly.

He then reviews why Postgres became the most popular database: it's packed with features, gives developers a great "day 0" experience, and has spatial data, vectors, JSON, and other features built in — it holds up almost no matter how you design your schema at the start. That makes it the database engineers reach for almost by default when they start writing an app. But Postgres itself is a 40-year-old design; while some of its architectural decisions have "aged like fine wine" in terms of extensibility, they're actually quite dated and hard to manage from a cloud operations and maintenance standpoint — especially around high availability and storage management.

He traces database architecture through its historical evolution: it started with "software installed on a single machine" of fixed size, which made it hard to adapt when you had lots of data but little compute (archival workloads) or lots of compute but little data. It then evolved into disaggregated architectures with storage and compute split apart, letting compute and storage use different node types and scale independently instead of being crammed onto the same machine. But if this approach relies on a closed, proprietary storage system, it turns into a data silo, and you still have to handle disk failures, erasure coding, and other complexity yourself — essentially the same problem object-storage engineers (building something like S3) are solving, just at a smaller scale, which amounts to reinventing the wheel.

So the "next step" is to put data directly on object storage, letting every system share the same copy of data and handing responsibility for long-term durability to object storage. But object storage has one fatal flaw: latency is too high (on the order of hundreds of milliseconds on average), which is nowhere near good enough for the sub-millisecond reads and writes a transactional database wants. So their design is: logically wrap Postgres inside a movable, stateless VM, write the actual data to a layer of local SSDs (safe-keeper nodes), then batch-convert it into another format and write it to S3. On reads, a page-server layer caches whatever pages are needed — this both "smooths out" object storage's high latency and lets the cache carry the system through whenever object storage has a hiccup, achieving higher availability than the cloud storage API itself.

In this architecture, Postgres itself is modified as little as possible, preserving its original transaction semantics and compatibility, while all high-availability, storage, and multi-tenancy management is handled by external Rust services. Postgres runs inside a Lambda-like VM, is stateless, and can be freely moved around and scaled vertically up or down in CPU and memory, also serving as a security boundary (since Postgres has a lot of extensions and carries real security risk). This approach lets them deliver several cloud-native capabilities that matter enormously for agents: true scale-to-zero (dropping compute to 0 when there's no traffic), fine-grained automatic vertical scaling (adding CPU/memory to a VM directly and live-migrating it in real time), and "branching" at the storage layer.

The final section focuses on this "database branching" capability. Because they control storage, they can let a user or agent cheaply cut a branch off an existing database, run a schema migration or an experiment on it, and roll it back at any time. In traditional preview deployments, all previews typically share one database, which makes migrations extremely painful; with branching, you can open an independent branch for every PR or every experiment, test against real data, and never pollute the primary database. The Replit agent, for instance, opens a database branch for every single "turn" so it can roll back easily once execution finishes — which is also how they first hit their "thousand-branch limit," and later had to raise the cap to five thousand.

Overall, this talk is making the point that AI agents bring database workloads that are massively short-lived and bursty and demand rapid experimentation and rollback — workloads that traditional Postgres deployment models can't withstand. What's needed instead is a new cloud-native OLTP architecture that turns Postgres into something that can be started and stopped freely, auto-scaled vertically, and split into separate storage and compute, with a high-availability, high-performance storage layer built on top of object storage — only then can it truly serve this generation of agent-driven applications.
