Context Engineering at the Frontier

Linus Lee, Thrive Capital

✕

This video describes how Thrive Capital built two AI agents (Puck and Hobgoblin), using carefully designed data structures, retrieval, and tool orchestration to turn the firm's sprawling institutional knowledge into a queryable, actionable "AI operating system."

The talk is given by Linus Lee, Head of AI at Thrive Capital, on the theme of "context engineering at the frontier." He starts by revisiting the origins of information retrieval in the 1960s: back then, as computers and data were exploding, the key question was how to find "just enough" content within a mass of information — not too little, and not drowned out by too much noise. That way of thinking — ranking, indexing, recall and precision, iterative refinement — is, in fact, the very core of what we today call "context engineering."

Next he introduces Thrive's internal system, "Thrive One" — a company-wide knowledge base and work platform spanning investing, legal, IR, talent, product, and more. On top of it, they built a research agent called Puck, dedicated to helping investors answer questions like: a company's recent developments, past views on a given market, relationships with certain founders, and so on. These questions all involve large amounts of both structured and unstructured data.

He describes Puck's evolution as breaking down into a few chapters:

First, building hybrid search as a foundational tool. The system uses both BM25 full-text search and vector search at once, then hands the results to a neural reranker to re-rank them. In practice, they found that for investing use cases, questions are often packed with "entities" like company and people names, and BM25's contribution can actually be even bigger than vector search's.

Second, pre-structuring and enriching "important entities" ahead of time (pre-enrichment). A lot of questions are really about a small handful of core companies and people, and the answers to those questions carry a kind of "socially / organizationally agreed-upon" definition — for instance, whether an "investment date" means the signing date, the wire-transfer date, or the internal decision date. Leaving the model to work this out from raw documents alone burns tokens and easily drifts from the organization's canonical answer. So at indexing time, they use a research agent to pre-assemble "authoritative" entity cards (with an explicit schema), triggering entity refreshes off signals like a new email, a new calendar invite, or a new document. That way, at inference time, the model sees a version of the world that's already been curated through a human lens, rather than raw, unfiltered noise.

Third, wrapping complex tools into sub-agents to avoid polluting the main model's context. Take querying a complex SQL schema: what the main model really needs is just the final results table, but if the SQL tool is used directly inside the main loop, the model ends up carrying the entire schema around and probing multiple tables, filling context with intermediate noise. So they have the main agent hand off only a higher-level question to a dedicated "SQL sub-agent," and the main agent only gets the query and its results back at the end. The upside is that the main context stays cleaner, and each sub-agent is easy to eval on its own; the downside is that the main agent and sub-agent end up playing a game of "telephone" — for very fine-grained, imperative workflows, instructions easily lose detail as they get relayed along the way.

Fourth, using "parallel sub-agents" for tool-choice routing. Some questions call for "completeness" rather than "find one answer that's good enough and stop" — for example, "what interactions have we actually had with this company?" Handed to a single agent, it will either stop too early (assuming the first hit is enough) or drag on for far too long. Thrive's approach is: the main model invokes a whole group of sub-agents at once, each specializing in a different data source or tool; each sub-agent decides for itself whether and how to use its tools, then reports its own results back for the main model to aggregate. The benefit is being able to "search globally" across multiple tools in parallel, improving both recall and latency; and as frontier models keep getting stronger, the quality of this kind of multi-agent collaboration keeps improving too.

Fifth, to traverse a large volume of documents without blowing up context, they designed a custom tool called "Snip." The scenario is something like: the model has to read an extremely long investment memo or a large batch of documents, but investors sometimes need precise, verbatim quotations (a client's exact words, fine print in financials, etc.) rather than just a sub-agent's summary. Snip has the sub-agent wrap "quoted excerpts" in its output with a special marker, and the system, in a post-processing step, strictly checks whether that text matches the source document word for word; if it does, the marker is kept so the main model knows this is a trustworthy quotation, and if not, it's stripped out. The main model's context ends up as a long string of highly condensed "source highlights," rather than secondhand summaries — preserving a high signal-to-noise ratio while retaining a precise, verifiable "view source" capability, at the cost of some extra tokens and compute.

The throughline of the first half of the talk is one idea: push complexity as far as possible into the data-structure and indexing stage, rather than dumping it onto the query-time prompt to muscle through. This echoes the old saying that "good programmers care about data structures and relationships" — in context engineering, that means leaning on schemas, entity graphs, and pre-enriched nodes to carry the burden of "modeling the world," so that queries can be both fast and precise.

In the second half, he introduces the second agent: Hobgoblin. Where Puck leans toward research and answering questions, Hobgoblin is a "knowledge worker agent that actually gets things done," built on an architecture similar to a coding agent. When Hobgoblin starts up, it's dropped into a custom Docker container holding a set of CLI tools for safely accessing various internal and external services: Thrive One, Puck's retrieval tools, documents, presentations, spreadsheets, the task system, the data warehouse, and even trusted external websites. Every request leaving the container has to pass through authorization and filtering by a network proxy; meanwhile, the specific user's credentials and a snapshot of their files are injected into the container, ensuring it can only see data consistent with that person's permissions, and sessions can be paused and resumed.

All of these CLIs are JSON-in / JSON-out, and Hobgoblin can only invoke them through shell commands — composing workflows out of the Unix toolchain just like a real engineer would. It wraps every internal tool behind a single unified CLI called "broom" (the author describes it as an internal version of MCP), then chains various tools together programmatically — for instance: call broom to pull back a Gmail thread, process it with a Python script, write it to a file, then feed it to the next tool. In this way, Hobgoblin becomes a little sprite working in a secure sandbox, "carrying a broom" around to sweep, tidy, and manipulate data.

Finally, he shares a real usage number: within a team of only about 80 people, over three thousand Hobgoblin tasks were kicked off in a single week — far more than Puck — showing that once an agent can genuinely "get things done," its value and usage frequency rise dramatically. He concludes that we're still facing fundamentally the same problem as in 1968: information overload, signal-to-noise ratio, and how to represent data so tools can use it well. The only difference is that today we additionally have large models, agents, sub-agents, and all kinds of new interfaces — while "retrieval and context design" remains the core of it all.
