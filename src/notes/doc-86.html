Towards Reliable Financial Agents: How a 4B Model Outsmarted a 235B Giant

Snorkel AI (with UC Berkeley)

✕

This video introduces how Snorkel AI, working with UC Berkeley's Sky Computing Lab, used a 4B expert model fine-tuned with reinforcement learning to beat a 235B general-purpose giant model on real financial-report analysis tasks — the key wasn't reasoning power, but "tool-use discipline" and reliability.

The speaker first explains Snorkel's research background: they focus on data-centric AI, and have published many benchmarks and open-source tools, of which this work is one. The collaborating team used the open-source RLLM framework to train specifically for the "financial Q&A + tool use" scenario.

The first step of the core experiment was to build the FinQA benchmark. The process has three stages: first, extract close to 7,000 SQL tables and schemas from SEC 10-K annual reports using models and programmatic methods; then, with the help of financial experts, design question types (growth rate, financial ratios, year-over-year change, etc.), and have the model automatically generate question-answer pairs plus structured intermediate information (which fields, rows, and reasoning steps were used) from the tables; finally, apply three layers of verification — programmatic checks (to avoid hallucinated fields/tables), automated review by a proxy model, and item-by-item correction by human experts — to obtain high-quality training and test data.

In this FinQA environment, the agent is given only four simple tools: list table names, look up table info, execute a SQL query, and a calculator. It sounds simple, but large general-purpose models (using Qwen3 235B as the example) make serious mistakes with these tools: they "hallucinate" table names out of thin air instead of using the available get_table_names; they like to fire off careless ‎⁠SELECT *⁠ queries that flood the context with useless data; and when they hit an error, they repeat the same failed strategy instead of correcting based on the error message. These issues also show up in other enterprise scenarios (e.g. insurance underwriting) and in closed-source models (like ChatGPT and Claude).

They then used the RLLM framework to RL fine-tune a Qwen3 4B Instruct base model. The environment is a classic reinforcement-learning loop: the model calls tools and queries SQLite (an in-memory set of 6,900 tables) over ReAct-style interactions of up to about 20 steps, and is finally rewarded via a binary correct/incorrect signal from an "LM-as-a-judge" (GPT-5 Nano). To overcome judge latency, they generated rollouts with roughly a thousand concurrent environments and cached judge calls via Portkey — about 40% of judge calls hit the cache — letting more than 360,000 scoring calls complete on 8 H100s in about 21 hours for under $500.

The training result: this 4B model reached close to 60% pass@1 on the FinQA benchmark, while the 235B version scored only about 51%. In other words, in this well-defined tool environment, the small, specialized model is not only cheaper but also more accurate. Even more interesting: although it was trained only on tasks with "a single table and a unique numeric answer," the "tool-use discipline" it learned transferred to the much harder FinQA Reasoning task — multiple tables (2–5) requiring an open-ended answer with analytical narrative. Here too, the 4B model significantly outperformed both its own baseline and the 235B model.

They also tested "cross-domain" effects by dropping the fine-tuned model onto the Berkeley Function Calling Leaderboard. The results showed that while there wasn't a big improvement on other general tool-calling tasks, there was no regression from the financial-specialization fine-tuning either — only small improvements on metrics like multi-turn memory — indicating that this kind of specialized RL fine-tuning doesn't destroy the model's original general tool-use ability.

On the training design side, the team ran several ablation studies. The first was on data curriculum: single-table data only, a mix of single-table and multi-table data, and a curriculum that starts with single-table before moving to multi-table. The results showed that once the model truly mastered the three "foundational tool skills" — schema exploration, correct SQL syntax, and error recovery — it could naturally compose them into multi-table capability; the real bottleneck was tool-use reliability itself, not simply feeding it more complex examples. The second was on reward design: comparing pure binary reward, partial credit (giving 0.2 just for querying the right table), and a fine-grained rubric (scoring data extraction, computation, reasoning structure, etc. item by item). The conclusion was that the simplest "0/1 correctness" reward actually worked best, because a complex reward too easily bakes in human bias about "the correct steps," constraining the model's exploration and ultimately suppressing performance.

The video closes with a few practical takeaways. First, for work with a highly fixed process and well-defined tools, like financial-report analysis, "efficiency can beat scale": a 4B expert model's inference cost is roughly 1/10th that of the 235B model, and an estimated 50,000 analysis queries per month can be handled on a single GPU — very favorable economics for enterprise deployment. Second, if your business process has clear tools and fixed steps (finance, healthcare, law, insurance, etc.), building a FinQA-style benchmark plus RLLM-style RL fine-tuning is a reusable blueprint. Third, Snorkel is investing several million dollars through its "Open Benchmark Grants" program to keep opening up more benchmarks, data, and model weights, hoping the community can make "reliable expert agents" — rather than giant generalists that just reason haphazardly — the new normal in real enterprise settings.
