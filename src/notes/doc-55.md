---
title: Let's Teach Claude Code Semantic Code Search With turbopuffer
speaker: turbopuffer
video: https://youtu.be/6t29UHJQpvA?si=9ucQlZvLJRJIhZDf
---
This talk explains how to equip Claude Code with a semantic code search system, implementing it with turbopuffer as a vector-plus-full-text search database and measuring the effect.

The speaker starts by clarifying a few terms: people often treat RAG as meaning "vector search only," but he stresses retrieval can actually be vector search, full-text search, or plain filtering; agentic search, likewise, isn't just grepping wildly — it means giving the agent a diverse toolkit so it can switch tools, change queries, and repeatedly narrow the scope when it can't find something.

He then contrasts Cursor and Claude Code. Cursor dumps every codebase (branch) into turbopuffer and does code search with "semantic search + grep," even designing a Merkle-tree-like structure so embeddings are only recomputed for files that changed on a branch, avoiding redundant cost. According to Cursor's internal experiments, adding semantic search lifts answer accuracy by about 12.5% on average, and by about 23.5% on their Composer model, which is specifically tuned for semantic search; real A/B tests also showed higher code retention and fewer unsatisfied requests on large codebases. By contrast, Claude Code currently relies only on grep-style tools, with no vector search.

He puts forward the idea that "embeddings = cached computation." If every session and every agent has to grep and read files from scratch each time, the same understanding work and token cost get repeated over and over; but if you first chunk, embed, and index the code into turbopuffer, an agent can afterward get relevant context quickly through a very cheap query.

On the implementation side, they built a simple CLI: it chunks the code, embeds it with Voyage's code2 model, and writes it into turbopuffer, then gives Claude a search tool that can answer natural-language questions like "how is password reset handled," returning the corresponding code snippet. The speaker specifically stresses that the chunking strategy is actually the hard part: ideally, "an unfamiliar but skilled engineer looking at just this one chunk could tell what it does, why, and how" — for that reason, some companies use an LLM to auto-generate documentation for uncommented functions before embedding them, to improve semantic quality.

To evaluate whether wiring this into Claude Code actually made things better, he used the public ContextBench benchmark. Rather than looking at whether the task ultimately gets solved, this benchmark checks whether the agent reads the human-annotated "golden files/golden lines" while solving the problem. He picked 50 questions that don't directly mention file names or function names, and compared three setups: vanilla Claude Code; Claude Code plus windowed grep (reading at most 50 lines at a time); and windowed grep plus a semantic search tool.

Along the way he found something interesting: Claude really doesn't like using new tools. Even after he added documentation and hints for when to use semantic search, Claude still instinctively grepped wildly and read whole files. Forcing it in the system prompt to "always use semantic search first" actually made performance worse, because semantic search isn't the best tool for every query — the key question becomes when to use which tool.

In the data, precision — how much of what gets read is golden files/lines — did improve markedly thanks to windowed grep and semantic search: vanilla Claude sits at about 65% precision, and adding windowed grep plus semantic search brings it close to 90%. Intuitively, that means vanilla Claude wastes tokens reading irrelevant files about a third of the time; adding windowed grep brings that down to a fifth; adding semantic search on top brings the share of "unnecessary context" read down to about one-eighth.

But recall — how much of the golden files/lines that should be seen actually get seen — isn't nearly as impressive on average: vanilla Claude reads so much that its file-level recall isn't too bad, but it drops off sharply once you get down to the line/symbol level; windowed grep and "windowed grep + semantic search" show roughly similar overall recall, with the semantic version even slightly lower on some metrics. The speaker further splits tasks into "semantic search does better" and "grep alone does better" and finds a clear division of labor: when you need to find "modules/files with similar behavior but different names," semantic search wins decisively; when you need to trace precisely along imports and function names, grep is actually the better tool. This shows the two excel at different types of tasks, and the hard part is training an agent to pick the right one.

The final takeaway: semantic search does improve precision and reduce needless file reads, but the real power is in the combination of tools — grep and semantic search each find different, complementary pieces of code. The long-term winning condition is giving the agent a diverse set of search tools (vector, full-text, regex, attribute filters, etc.) for finding context, and, like Cursor, using methods like RL to train the model to learn which tool to pick in which situation, instead of forcing a single approach through prompting alone. He closes by noting that the simple CLI and example code from this experiment are open-sourced in the Turbo Grep project, and that a more complete guide on "how to do code search on turbopuffer" is coming later.
