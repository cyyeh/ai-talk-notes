---
title: The Modern Data Stack Lost the War: Stop Building more DataFrame APIs
speaker: OpenAI
video: https://youtu.be/xp1wC3nx94Q?si=52ziNVtZ5Oiguk0B
---
This video argues: stop putting faith in yet another new DataFrame API, and instead adopt "function-centric data tools" to serve AI agents.

The speaker starts with the evolution of data tools: from spreadsheets, to the data engineers/data scientists of the 2010s raised on frameworks like Pandas, to today's AI agents driven by natural-language interfaces. Many companies today are building "data agents": prompt plus context, handing things to a model that calls a code-interpreter-like tool to produce results. But using a "10 + 32" example, he points out that running the full agent pipeline is actually slow and expensive — far less efficient than a human just writing a small snippet of Python and running it locally.

He then reviews why humans love data frames so much: the tabular structure is intuitive, the API is composable and readable, and it's far easier to understand than complex SQL, letting us process data iteratively. But across the worlds of Pandas, Polars, and PySpark, everyone is really doing very similar things — just with different API styles and performance characteristics. In the past, "which API feels more human" was the deciding factor; but in the LLM era, the readability advantage of an API erodes, because models can generate and translate these APIs themselves.

So if you're going to build yet another new DataFrame library today — whatever kind, a "video dataframe," a "geo dataframe," or "rewritten in Rust" — the focus shouldn't be what the API looks like, but rather: are we using the correct abstraction? His proposed answer is to extract the core logic into simple, reusable Python functions (user-defined functions, or UDFs).

He points out that traditional data frame systems mostly bolt on UDF support as an afterthought, with optimization concentrated on the framework's own column expressions, so performance collapses as soon as you rely heavily on UDFs. There's a benchmark experiment in the video: using 12 UDFs to process a 10GB CSV on a standard machine, comparing Pandas, PySpark, and Polars on single-threaded versus multi-core performance. The conclusion: some frameworks scale with core count and some don't, and "plain Python code" sometimes even beats these frameworks' performance in UDF-heavy scenarios.

He argues that redesigning the engine from a "function-first" perspective can massively optimize performance while preserving ordinary Python syntax and function style: single-threaded execution alone can be an order of magnitude faster than plain Python, and it can scale effectively and near-linearly across cores — the demo numbers go from roughly 144 seconds down to 26 seconds on a single core, and then to about 4 seconds across multiple cores. In other words, truly treating functions as first-class citizens, combined with a good query engine, can give you a full order-of-magnitude performance advantage over the existing data-frame ecosystem.

Going further, once the core abstraction becomes "Python functions + a UDF engine," you can keep the same API while swapping in different execution engines underneath, or even lean on a large model to automatically generate a "translation layer." He shares an experiment: using the existing API plus a UDF-first engine, then spinning up a few Codex instances and designing a simple eval, the model produced, within a single day, a translation layer that used to take the industry 20 people and 2 years to build. This shows that focusing on the right abstraction (functions + engine) can let AI "rewrite" massive amounts of infrastructure for you.

In summary, his conclusion is: in the AI/agent era, data tools should shift from "yet another new DataFrame API" toward a data-oriented code tool built on "high-level operators + shared Python functions + an efficient UDF engine." Functions are like Lego bricks for agents: composable, testable, evaluable, reusable across an entire organization, and able to win on both performance and productivity through a powerful underlying engine plus automatic AI translation. He closes with a gaming metaphor: this kind of data-centric code tool doesn't replace the large model — it's more like the "mushroom" in Super Mario, a power-up that makes agents stronger while saving money — and he calls on everyone to "stop building more DataFrame libraries, and start building data-centric code tools for agents."
