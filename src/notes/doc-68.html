Running Millions of (Millisecond) AI Sandboxes without Breaking the Piggy Bank

Felipe, Unikraft

✕

This video shows how Unikraft uses extremely optimized virtualization technology to run "millions of, millisecond-startup" AI sandboxes/agents without blowing up your cloud bill.

Felipe starts from the fundamentals: the architectural differences between VMs, microVMs, unikernels, containers, and runtime isolates (V8, JVM, etc.), focusing on "isolation strength" and the size of the TCB (Trusted Computing Base). A VM only shares a relatively small hypervisor (e.g. KVM, at a couple hundred thousand lines of code), whereas a container shares the entire Linux kernel plus the container runtime — often tens of millions of lines — making the odds of a vulnerability much higher; stacking V8/JVM on top of that makes it even heavier. So he emphasizes: for multi-tenant workloads, especially AI agents you "can't trust," containers actually aren't secure enough — the true gold standard in public cloud is still the VM, and a lot of cloud providers actually sneak a VM layer underneath their containers anyway.

He then walks through his own research history: he started out building high-speed packet-processing VMs using a highly customized, very small OS (a unikernel), proving that a VM can be both extremely fast and extremely small — you don't have to choose between performance and security. They later published papers like "My VM is Lighter and Safer than your Container," demonstrating startup times of a few milliseconds and thousands of VMs running on a single machine, and turned it into a Linux Foundation open-source project and SDK for automatically generating this kind of purpose-built VM (unikernel), instead of hand-building one each time.

Later on, they expanded the goal from a "single VM" to the "entire cloud platform": not just the VM itself needing millisecond startup, but the whole chain — load balancer, proxy, controller, VMM (e.g. Firecracker) — all needing to complete the "sleep to serving a request" wake-up flow within 10ms, so that users never notice the VM was ever scaled to zero. In fact, they initially just converted a Dockerfile into an AMI and threw it on AWS, only to find startup took 30 seconds — that's when they realized the components surrounding the cloud platform were the real bottleneck.

To solve the problem of slow application startup itself (like a JVM taking several seconds just to spin up), he introduces techniques like snapshot, fork, and checkpoint: let the VM fully boot and warm up the JVM or other heavy runtime, then take a snapshot of that state, so that every subsequent instance can restore from that snapshot within milliseconds. Snapshots can also be chained into a timeline, letting you later "rewind" to any checkpoint, or branch off parallel VMs from it. Fork lets a running VM "spawn a child VM" without interruption — great for an agent spawning sub-agents, or a database branching off experimental copies — and each child is still a full VM, with isolation fully intact.

In the agents/sandboxes scenario, these capabilities all come together:
Agents fundamentally need to run inside a complete VM/sandbox to be safe, and their behavior is intermittent — they compute for a bit, then call a browser or a tool, then sit idle for a while. So the best approach is: scale the corresponding VM to zero whenever the agent goes idle, using a snapshot to preserve all state; when a request comes in, restore from the snapshot within milliseconds, and the agent never even knows it was asleep. This lets you pack an enormous number of "dormant VMs" onto a single machine — in actual testing, they packed over one million "wakeable" VMs onto one ordinary 48-core server, waking them on request and putting them back to sleep once they'd responded.

He also mentions that reaching this kind of scale required working through a lot of Linux/KVM pitfalls: Firecracker looks like a plain process to Linux, and once you have tens of thousands, hundreds of thousands, or even over a million processes, things like tap devices and the network stack start to buckle; even software like Tailscale can crash from scanning too many tap devices. So they made a number of low-level engineering adjustments to keep the system stable at this "atypical" scale.

In the final demo, he uses the Unikraft CLI to spin up an Nginx VM on a roughly 24-core server, configured with a "scale to zero" policy. When there's no traffic, the VM sits in standby (asleep, consuming no resources); the moment an HTTP request comes in it wakes instantly, handles it, and immediately drops back to standby — the whole process feels almost latency-free. He also demos a single machine running around a thousand "Open-Claude-like" chat VMs, each with its own state; he can wake a specific numbered instance, resume its earlier conversation, kill it and bring it back, or use a script to wake a whole batch of VMs in sequence, watching them wake up "snake-like," respond, and drift back to sleep one after another.

The video's overall conclusion is:
When designing large-scale AI agent/sandbox infrastructure, you don't need to make a painful trade-off between "speed, scalability, and security isolation." As long as you get the design and engineering optimization right — from the VM/hypervisor all the way through the proxy, controller, snapshot mechanism, and OS layer — you can simultaneously achieve millisecond-level startup, million-scale VM density on a single machine, and multi-tenant isolation far stronger than containers, letting you run millions of AI agents in a single rack instead of an entire data center.
