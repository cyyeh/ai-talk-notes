Five years of OpenLineage: How we built an industry standard and why agents need it

Datadog (OpenLineage)

✕

This video explains what OpenLineage is, why we need a "standardized language for data lineage," and how it became an industry standard that matters even more in the era of AI agents.

The talk opens by explaining data lineage: metadata about a data pipeline — where the data comes from, where it goes, and what transformations happen along the way — which helps people understand and reason about the overall data flow. [0:15][1:18] Today's data ecosystem is highly fragmented: different clouds, different platforms, different frameworks, different tools, each speaking its own format and language, which makes it very hard to build a consistent lineage across the whole environment. [0:31][0:49]

He compares this to mismatched plug standards: you bring your plug to another country, and it's still technically "carrying electricity," but it doesn't fit the wall socket, so it's useless. [0:49][1:06] For tools to truly interoperate, you need a shared "standard language" to describe data lineage.

Next, he outlines three common approaches for observing lineage: [2:15][2:36]

  1. Analyzing source code — inferring inputs, outputs, and relationships from the pipeline/job's code.

  2. Parsing activity logs or query history — deriving lineage from actual execution records.

  3. Observing and collecting metadata while the pipeline runs (runtime observation) — recording the facts directly as they happen.

OpenLineage takes the third approach — "collecting lineage events at runtime." [2:44][3:17] He illustrates this with a photography example: you could look at a photo afterward and guess the time, location, and lighting, or you could just look at the metadata the camera recorded automatically — the latter is far more accurate. [2:55][3:13] In the same way, runtime metadata describes what actually happened in a data pipeline much more precisely.

He then formally introduces OpenLineage: it's essentially a specification that uses JSON to describe lineage events; it's vendor-neutral, lives under the Linux Foundation, and provides shared libraries for common frameworks plus integrations with various data tools. [3:15][3:38] The key value of this "standard language" is that you no longer need to write a point-to-point integration for every platform — you can rely on a common event format, and only need to know how to "produce and consume" this standard event, drastically cutting down on repetitive boilerplate integration code. [3:40][4:15]

The specification has a few core concepts: [4:18][4:36]

  • Job: an abstract entity that does something (e.g., an ETL task).

  • Run: a concrete execution instance of a Job.

  • Dataset: the input and output datasets of that run.

All of these can be extended through facets, making it easy to attach extra metadata for different scenarios without breaking the shared core semantics. [4:36][4:45] And because there's a consistent way of describing things (for example, how a Postgres table's location is denoted), information from different telemetry sources can be "matched up" to assemble a complete lineage picture. [5:07][5:33]

He then reviews the project's history: back in 2018 at WeWork, they needed a metadata store internally and couldn't find an off-the-shelf product, so they started the Marquez project. [5:38][5:59] From Marquez's practical experience grew the idea of a fully vendor-neutral specification decoupled from any implementation — which later became OpenLineage. [5:57][6:06] They built a handful of initial integrations, and the community picked up from there; today there are roughly 40 producers/consumers, adopted by several major cloud vendors. [6:14][6:24]

He emphasizes that a standard like this doesn't grow out of a "my solution is better, everyone should use it" approach. Instead, it comes from finding a group of people who care about the same problem, identifying the common ground across different parties, and designing things so every role has an incentive to participate and win together, reducing competition and friction. This incentive-alignment approach is what gradually produced today's "snowball effect." [1:44][2:00][6:25][6:41]

Finally, he brings the discussion back to AI agents. More and more agents are now writing code, building pipelines, and moving data, but the user themselves may not actually know what data the agent is reading and writing, where it comes from, or where it goes. [7:00][7:08] In this situation, lineage and OpenLineage become even more critical:

  • You need an observable, traceable system before you can trust that an agent is operating on the right data. [7:11][7:16]

  • To reproduce a given process, you need auditable, cross-platform, reproducible system records. [7:19][7:27]

To him, lineage metadata is the key infrastructure that turns an agent from a black box into something with a contextual knowledge graph.

He wraps up the video briefly: the OpenLineage community welcomes new members, who can join via their Slack or other channels; if you're interested in what he's working on at Datadog, you can look into that separately. [7:29][7:46]
