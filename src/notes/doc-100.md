---
title: Learning while you sleep: Beyond memory to dreaming
speaker: Lamis Mukta, Anthropic
video: https://youtu.be/tTcxVv8HHNw?si=58gPrZDf8w_rFfFp
---
This talk traces the evolution from "memory systems" toward "dreaming" — an architecture that lets agents genuinely keep learning — and argues that a smarter model alone can't get you there.

The starting premise is that raw model intelligence isn't enough to build a durable, scalable product; the real lever is context engineering — feeding an agent the specifics of your organization, codebase, and user preferences so it becomes useful inside your environment and keeps improving over time. The talk recaps a year of Anthropic's progression on context and memory. It began with a simple Claude MD file — a long markdown guide injected at the start of every session — which is remarkably effective at steering behavior but bloats over time and eats into the context window. Next came memory tools that let the agent decide for itself when to read and write memory, bringing autonomy and a real sense of continual learning, but also the first scaling problems. Then Skills, which use progressive disclosure: the agent reads only the first few lines of a file as a table of contents and loads the detail on demand, like pulling a book off a shelf — great for well-defined, process-shaped tasks, but still human-driven, since a person decides what becomes a skill. The current state of the art models memory as a file system: everything is a markdown file, the agent reads and writes with bash-, grep-, and index-style search, and a good index drives the progressive disclosure.

A file-system-style memory store that the agent manages autonomously looks like continual learning — but in real production it hits several walls. Multiple agents may write to the same memory file concurrently and conflict. One agent writing something wrong into shared, org-wide memory amplifies the error to every agent. Humans and agents both edit memory, so you need to track who changed what and on what basis. And memories go stale, turn out wrong, or can even be maliciously injected.

Those pressures yield a set of design principles for production-grade memory. Versioning: every update carries a version, its source session/transcript, and who (which agent or person) made it, enabling rollback and audit. Concurrency control via hashing: read the memory and record its hash, draft an edit, then recompute the hash before writing — if it changed underneath you, the write is refused and you re-read and re-edit. Permissioning: org-level knowledge may be read-only while a personal scratchpad is writable, with different visibility and write scopes per group or product line. And portability: memory should be reachable through a clean API so it can be reused across products and systems rather than locked into a single tool. Designed this way, memory yields more accurate, cheaper behavior across multi-agent, long-running, complex domains — because next time the agent can get it right in one shot instead of reasoning from scratch.

The talk then names two structural limits of in-band memory (memory maintained by the same agent doing the task). First, resource contention: a single agent in a single session must both do the current task well and curate memory for its future self, a hard optimization over how many tokens to spend on the task now versus on paving the way for later. Second, a narrow horizon: a single-session agent can't see cross-session failure patterns (the same bug recurring) or the performance of the whole agent fleet (where other agents keep tripping in other customers' environments). Memory also needs active checking and pruning, and an in-band agent busy with its task rarely stops to do the housekeeping.

These limits motivate dreaming — an out-of-band, batch, asynchronous process dedicated to reviewing all the agents' behavior and reorganizing the memory store. It takes the existing memory (a set of markdown files), collects a large batch of agent-session transcripts over some window (conversations, tool calls, which skills were used, and other metadata), and spins up a dreaming agent — really an orchestrator over a fleet of sub-agents — that looks for patterns where agents repeatedly failed or underperformed, checks them against the current memory (what knowledge is missing, wrong, or outdated), and proposes a new memory store: additions, edits, and deletions, each annotated with where the problem showed up, how frequently, and why it's worth changing. The orchestrator consolidates the sub-agents' suggestions into the genuinely representative patterns and emits a reviewable change list that the product team can accept or reject.

The analogy is a school: the working agents are the students, the dreaming system is the teacher or principal, and the memory store is the curriculum. A teacher doesn't just grade individual homework — they look across a whole cohort's performance to revise the curriculum, filling gaps and correcting common misconceptions. Because dreaming runs out of band, it doesn't compete for the task's context or token budget, it can reason across sessions and across the whole fleet, and it actively deletes stale content and adds what's missing to keep memory fresh.

The system is ultimately two loops running in parallel. Memory (in-band) is the short loop: a single agent writes valuable information as it goes, so the next run of that agent — or that user or project — starts stronger. Dreaming (out-of-band) is the long loop: a separate batch process parses large volumes of transcripts, error logs, and tool-call history, driven by an orchestrator that points sub-agents at the dimensions you care about (error rate, completion time, style consistency) and returns memory-update proposals, spending its tokens on making all future agents better rather than on any one task. Dreaming costs extra tokens and looks more expensive up front, but because it lets agents get later tasks right in one pass with fewer detours, it can lower overall cost and latency.

The closing recommendation is a three-layer path. Start with the simple, effective things — Claude MD, skills, and basic memory tools already sharpen an agent's usefulness, and not just for code: slides, writing style, and personal preferences all benefit. When you have many agents, long-lived codebases, or a complex domain, add production guardrails — versioning, concurrency control, permissioning, and portability. And to truly reach continual learning, add an out-of-band dreaming layer that periodically prunes, reorganizes, and expands the memory store so agents get smarter, cheaper, and faster over time. The whole arc from context engineering to memory to dreaming has mostly emerged in the past year — an open, fast-moving area with much more implementation and research still to come.
