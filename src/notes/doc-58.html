Malleable Evals: Why Are We Evaluating Adaptive Systems with Static Tests?

Vincent Koc, OpenClaw

✕

This talk is about why evaluating AI systems that keep changing with "static" evals is no longer good enough.

The talk opens with Vincent starting from the traditional software world: we rely on unit tests, regression tests, CI/CD, plus a bit of chaos engineering and observability to keep systems stable. [1:05–2:48] But when this playbook gets carried over to AI/LLM applications, most people stop at "static benchmarks" and hand-built question sets: manually design a batch of questions, test offline, and if the model doesn't say anything that violates the rules, ship it. [2:52–3:33] The problem is that AI applications, and the contexts they're used in, aren't static at all — yet we're still treating them like ordinary software.

He reviews how LLM development has changed in recent years: early on it was "prompt engineering" — people frantically swapped prompt wording, like blindly trying medications, keeping whatever happened to work. [5:37–6:23] Then came the "context engineering" era, with RAG and tool calling, breaking a large agent down into many small, testable tool modules, and eval started becoming more tractable. [6:27–7:16] From 2025 onward, models got more capable, tokens and the cost of generating code got cheaper, software iteration speed exploded, and it even became possible to use an LLM to solve hard ARC-I2 type visual/pattern problems. [7:22–8:25]

Against this backdrop, he proposes "intent engineering": the machine self-optimizes based on "intent" — for example, modern harnesses (like OpenClaw, and various agent frameworks) try to understand and adapt to you. [8:27–8:49] But as soon as the experience differs from person to person, organization to organization, eval gets much harder: your path and my path are completely different, and the traditional model of "one question set conquers all" simply can't capture real behavior. [8:51–9:14]

So he pushes back on the "evals are dead" claim — he argues the opposite is true: in agentic applications, we need, even more, a multi-layered understanding of what's actually happening inside the agent; it's just that the form eval takes needs to change. [9:15–9:31] The traditional pipeline is: static benchmark → hand-built eval dataset → offline evaluation → after launch, a lack of chaos/real-behavior observation, so when something goes wrong you're back to square one, redoing everything. [9:43–10:02]

He then proposes a new direction: shift eval from "matching the correct answer" to "outcomes aligned with intent." Instead of just asking "does 1+1 equal 2," ask: in an ambiguous, multi-step, multi-tool situation, does the agent match the "style, persona, fuzzy criteria" we actually want? [10:11–10:22] On the research side, you can describe these fuzzy criteria using a rubric (the way you'd grade an art assignment). [10:31–10:39]

He goes on to discuss a few key concepts:

  1. Self-curating test sets from traces: in practice, 80% of usage scenarios repeat in a stable way, but as the user base changes, the shape of questions gradually drifts. [10:50–11:04] If we capture actual execution traces and let the agent build or update its own test set from those traces, it can keep reflecting the latest usage behavior, instead of forever testing a new behavior space with an old dataset. [11:04–11:16]

  2. Online, always-on eval/optimization: eval is no longer just a one-time, pre-launch action — it's an "always-on" service that continuously watches traces, continuously runs agent-based eval, and continuously adjusts. [11:19–11:29]

  3. Telemetry in the loop, a self-healing harness: he mentions his own paper concept — if an agent's harness is aware of telemetry (error rate, cost, latency, etc.) and has some conditions in place (budget, quality thresholds), it can self-adjust and self-repair the moment it detects an anomaly, instead of waiting for a human to step in. [11:34–12:00] This makes the system more like a self-healing agent, rather than a rigid pipeline.

He calls this whole phenomenon "eval calcification": if our evals stay stuck on a fixed dataset and a fixed process, while actual agent and user behavior become faster-moving and more variable, eval itself will calcify — falling further and further behind reality. [12:12–12:27]

As a vision, he cites an idea similar to Karpathy's on auto-research/auto-optimization: you only need to define the goal and the reward signal, and the system explores the optimal configuration in parameter space on its own. [12:31–12:59] For AI applications, eval is no longer a "dataset starting point" — it's the "end state I want to reach" — such as user satisfaction, error risk, or stylistic consistency — and then you let the machine continuously tune parameters and self-correct around that end state. [13:11–13:21]

Finally, he closes with an 80/20 metaphor: 80% of behavior can be covered by fairly static, clearly defined intent descriptions and test coverage, but what will actually wreck your business is that ever-shifting 20%: the weird questions, the unexpected usage patterns, the edge cases. [13:32–13:46] The key is to use an agent to manage that 20%, continuously observing and adapting, and to make eval itself a living agent — one that updates its tests based on traces and self-adjusts based on telemetry, rather than a fixed, unchanging benchmark package. [13:48–14:06]

Overall, the core thesis of this talk is: in the era of agentic, intent-driven AI systems, eval should be designed as "software/an agent that grows and self-optimizes alongside the system," not a static question bank.
