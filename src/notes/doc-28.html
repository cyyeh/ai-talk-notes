Context Engineering 2.0: Unifying MCP, Agentic RAG, and Memory

Redis

✕

This video argues that getting an AI agent to work reliably in the real world hinges not on the model, but on "context." Redis proposes an overall architecture called a "context engine," packaging RAG, tool calling, memory, and the semantic layer into a single layer of infrastructure.

Key concepts in the video

The speaker starts by noting that today's LLM agents can already reason continuously for an hour without breaking down, and this "continuous reasoning time" roughly doubles every six months. That means we're no longer just building chatbots or summarizers — we can build long-running agents. But the precondition is giving it "the right context."

He breaks agent capability down into four key dimensions, all related to context:

  1. Dynamically navigable context (not just RAG)
    Traditional RAG is linear: "look up the data for the model first, then dump it into the prompt in one shot."
    But an agent that genuinely "works" behaves more like an engineer writing code:
    it looks at one file, learns a bit, and then, based on that new information, decides which file to look at next — rather than loading an entire repo all at once.
    So context should be something "the agent can navigate on its own," not something a human prepares for it in a single pass.

  2. Context needs to be fast and fresh (low latency plus real-time)
    An agent calls a large number of tools frequently, and if every one of those calls slowly hits an external API (e.g. Google Calendar), the user experience falls apart.
    He gives the example of building his own EA agent: hitting the Google API directly was slow at first, but once the data was synced into Redis and the agent queried Redis via MCP instead, the speed became reasonable.
    Also, if you feed an agent stale data as context, it's almost guaranteed to hallucinate, so the system must be able to keep syncing the latest state continuously.

  3. From "RAG" to "Agentic RAG": RAG isn't dead, but it needs to become a tool
    He criticizes how a lot of current systems just "turn documents into vectors and use RAG to fetch a blob of text for the model" — an architecture that's linear and can't really tap into an agent's capacity for long-running reasoning.
    For example, a customer-service bot gets asked, "why is my delivery order delayed?"
    Traditional RAG: it just pulls a passage about "common causes of delay" out of a policy document and pastes it back to you.
    A genuinely useful agent should instead:
    go query the order data, the merchant data, the driver's location, and other structured data itself, then answer based on that actual state.
    So the "RAG tool" itself should become one of the agent's search tools, rather than having a human pre-fetch things for the agent ahead of time.

  4. Memory is actually "state," not just something for personalization
    He splits memory into short-term and long-term:

▫ Short-term: a thread like ChatGPT's conversation history, which needs mechanisms like queuing, compression, and summarization to keep context from growing without bound.

▫ Long-term: extracting durable facts and preferences out of conversations and behavior.
He isn't fond of the word "memory" — he feels it too easily gets reduced to personalization along the lines of "knowing you like spicy food."
In his view, what matters more is:

▫ Distilling the decision paths an agent has "walked" into a kind of state, so that next time it hits a similar situation it steps on fewer landmines.

▫ Running ETL-like extraction over all kinds of events, forming a compressed, searchable "internal context."
Redis offers a set of tools called Redis Agent Memory, supporting discrete facts, summaries, preferences, and "custom extractors" — using an LLM to extract state against a schema you define yourself.

Why relying on traditional RAG alone leaves you "behind"

He puts it bluntly: if all you're doing today is traditional RAG, you're already behind.

The reason is that the traditional RAG architecture is:

query once → feed the model → answer once

This fails to make full use of the fact that agents can now "reason and explore continuously on their own."
The system of the future should instead:

give the agent "a world of data it can explore (a semantic layer plus tools),"
letting it decide at any point in the problem-solving process "which table, which index, which data source" to query right now.

He calls this new way of thinking a "context engine" — RAG is just one tool inside it, not the whole architecture.

Redis's proposed "Context Engine" architecture

In the second half, he uses Redis's own product line to show how this context engine gets implemented (though in principle it generalizes to any tech stack):

  1. Fresh data / ETL layer
    Data from various sources (databases, APIs, event streams, etc.) flows into Redis via ETL, pre-organized into data structures suited for agent use.
    The key points are:

▫ Don't let the agent go run a large batch job directly (e.g. don't have it fire off a Spark job on its own).

▫ Data for most commonly used queries should be precomputed ahead of time, in a state that supports low-latency querying.

  1. Context Retriever / Semantic Layer (typically via MCP + Pydantic)
    They've launched Redis Context Retriever, which can automatically build a semantic layer on top of Redis:

▫ Use Pydantic to define the data model.

▫ Generate a corresponding MCP server / CLI, letting the agent explore data "through semantic understanding" rather than "memorizing endpoints by rote."
He stresses: if all you do is wrap an existing REST API in an MCP layer, you haven't solved the problem — you've just moved house.

  1. Memory system

▫ Short-term: conversation messages and agent traces are stored in Redis, with automatic summarization and compression.

▫ Long-term: a memory extractor pulls durable facts and decision rules out of messages, files, or databases, forming state.
He views this as a kind of "fuzzy ETL":
using an LLM to distill "what's genuinely important and durable" out of messy context, cutting down how much context each inference needs.

  1. Semantic caching, cutting LLM cost
    A lot of conversations are repeated questions — like "what are your hours?" asked in several different phrasings.
    They built a model fine-tuned from a modern BERT-based architecture, used to judge whether two queries are semantically equivalent, letting the system reuse an old response and saving up to 80% of LLM cost.
    This comes in both an open-source version and a commercial version (LangCache).

Demo: a DoorDash-style customer-service bot

He demos a "DoorDash-like customer-service bot" built on the context engine architecture:

  • Question: "Why is my order delayed?"
    The agent doesn't go look up a "delay-policy PDF" — instead:

▫ It uses the context retriever tool to find the order, restaurant, driver, and other structured data sources itself.

▫ It looks up the order's actual status and replies: "Your sushi order is delayed because the driver is currently at…"

  • Question: "What should I order next?"
    The agent will:

▫ Read the user's past orders and long-term memory (e.g. a preference for spicy food).

▫ Combine that with the current menu to make a personalized recommendation.

▫ And show exactly what's stored in short-term/long-term memory.

This demonstrates what he means by:

treating context as a "surface that can be explored,"
with the agent roaming freely across it through tools (e.g. MCP, the semantic layer), rather than a human pre-preparing RAG results for it in one shot.

The new role of "Context Engineer"

Finally, he predicts a new job function will emerge: the context engineer.

This kind of engineer is specifically responsible for:

  • Designing and implementing the context engine:
    data syncing, structuring, semantic-layer design, tool interfaces.

  • Handling the old problems of data and security, now applied to the agent world:
    permission control, prompt injection / data injection, and dealing with noisy data.

  • Combining eval (evaluation) with data quality, to make sure the agent works reliably in real-world situations.

He says that back when he worked on traditional MLOps, he already believed "data plus eval" was the hardest part, and the essence hasn't changed in the agent era — it's just taken on the concrete form of "context plus eval."

Summed up in one line

The core message of the whole talk:
a genuinely usable AI agent has to be built on top of a "context engine" — one that integrates fresh data, a semantic layer, memory, and caching, letting the agent explore, update, and use context itself through tools, rather than passively receiving RAG results a human prepared for it in advance.
