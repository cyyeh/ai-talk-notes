---
title: Powering Agents with Context Graphs & Ontologies
speaker: Datalinks
video: https://youtu.be/4Sakcqb3s0g?si=eYvdPHPuCA2tlUxX
---
This video introduces how Datalinks uses "tabular graphs" and ontologies to give AI agents memory, traceability, and a queryable context layer — a "context graph."

The speaker opens by noting that although everyone talks about graph data and ontologies, real-world enterprise data mostly still lives in tables, spreadsheets, data warehouses, and lakehouses. Traditional graph databases tend to hit walls on performance and complexity once node counts reach the billions, or when multiple graphs need to be stitched together. So their approach is to keep the scalability and familiarity of tables, but build links "cell to cell," connecting everything into one giant "tabular graph," and then layer ontology and graph semantics on top.

The overall pipeline works roughly like this: unstructured documents such as PDFs are fed into a parser and first extracted into raw tables, then a module called "auto model" simulates a data engineer — based on the user's prompt, it automatically identifies corresponding entities and fields such as "product, ingredient, specification, test result, marketing copy," produces a schema, and ends up with a clean wide table. Entity alignment and field-to-field linking follow, forming a cell-to-cell graph. This isn't limited to structured data — in their example, they also tabularized about 400,000 rows (roughly 2GB) of Reddit cosmetics discussion data and connected it into the same graph, to simulate a customer service scenario.

Because the core is still tabular, they can keep traditional analytics operations (filtering columns, group by, aggregation) while also being able to "hop" between tables — for instance, jumping from certain "complained-about ingredients" to the formulas containing those ingredients, then to the finished products, then to the Reddit conversations complaining about those ingredients. The agent uses a query language on this graph that resembles a general-purpose programming language, rather than a specialized graph language like Cypher/Neo4j — so even small models can understand, generate, and execute the queries, and at lower cost.

In the live demo, he used a cosmetics company as an example: first tabularizing 130 cosmetics formula PDFs, then connecting them to 400,000 rows of Reddit discussion. The agent can be asked in natural language to "find the ingredients people are complaining about"; the system first parses the question through the taxonomy, then automatically generates a query, and the result gives not just "which ingredients are complained about" but can also handle a follow-up like "find 5 users who complained about these ingredients and write an upsell message for each one, recommending our own products that contain these ingredients" — while the UI shows every query step, which tables it hopped to, and which rows and columns were used. The speaker emphasizes that this ability to "trace back exactly what specific data the agent used to produce its answer" is the embryonic form of a context graph.

Midway through, he addresses two key questions: first, "how to model the data," and second, "how to keep the agent's context from being drowned in tokens." He believes the bigger challenge is actually the former: how you partition data objects, define each table's columns, and decide which columns link to which — this determines whether the agent can reliably find the right data, rather than repeatedly falling into a loop of "sorry, I can't find the data." His modeling principle is: each table should be "sufficient on its own to build an executive dashboard," and column names should be semantically clear to both ordinary people and LLMs; tables should be "wide and somewhat redundant," but each table should have standalone value, connected into a graph through a handful of linking columns, preserving a structure humans can intuitively understand. That way, the agent only needs to see the schema plus link descriptions to plan its own queries and hops.

He then moves on to the recently popular concept of the "context graph": beyond having a "context layer" (the queryable knowledge graph described above), you also need a graph dedicated to storing the "agent's decision trajectory." This includes what questions the agent asked, what data it queried, what queries it ran, what intermediate reasoning it produced, and what actions it ultimately took. This matters greatly for heavily regulated industries (finance, accounting) or scenarios requiring accountability tracing, because companies want a clear "chain of decision evidence" for the agent during audits or when something goes wrong, shifting some of the responsibility away from individual people and onto an "inspectable system process."

In this section, he shows how to save an entire agent execution trace (questions, queries, results, reasoning text, etc.) as a text file and feed it back into the same auto-model pipeline, this time with a prompt along the lines of "I want to build a context graph that traces agent decisions." The system automatically proposes a set of objects (queries, reasoning_steps, ingredients, complaints, etc.); after you review them, it extracts these entity relationships from the trace and turns them into a new graph. This context graph becomes a "decision graph" that can later be queried — for example, "load all the data and reasoning used in this agent run, then present it in a more digestible form for a human reviewer to check." He notes that an undesigned raw trace is an information explosion for most users, and a well-designed UX and abstraction layer are essential to understanding what the agent is actually doing.

In the Q&A, someone asked how the ontology he uses differs from the traditional Semantic Web / OWL approach. He said he personally isn't fond of heavyweight semantic layers, and rarely sees OWL genuinely deployed in production outside academic teams. His definition of ontology is more pragmatic: it's simply a set of "interconnected objects," each useful in its own right, while also being searchable to find related neighboring entities. The formal logical axioms and taxonomy structure underneath aren't the key thing for most business users — what matters is being able to "search from a particular car instance all the way to its last maintenance record, then jump to the parts that were replaced," not knowing how many axioms are maintaining consistency behind the scenes.

Also in this session, someone asked: since the extraction and alignment process is itself stochastic, and links produced at different times vary slightly, how precisely do humans actually need to verify ontology relationships? Or should they just care about output quality? He replied that in practice, different people naturally take different data paths to solve the same problem, so their goal is less about pursuing a single, uniquely correct graph and more about "having the system propose links automatically at scale, then giving humans or agents tools to sample, adjust, and prune." Whether to let the graph evolve as new data comes in, and whether to auto-prune old links, has to be designed per specific application scenario — there's no universal best answer.

The last few questions centered on data lineage and standards such as OpenLineage; he admitted that Datalinks' lineage is currently in an internal format, but said they would look into these open standards. Someone also asked whether they perform entity resolution and indexing during lineage generation — he answered that they build an index for every cell, and these indexes underpin the cross-table hopping and trace queries shown earlier.

Overall, the focus of this talk is on using a "tables-first, graph-semantics-layered-on-top" architecture to solve three things at once: enabling agents to reason and retrieve reliably over real-world tabular data; letting both humans and agents collaborate on an understandable ontology and schema; and using context graphs to fully record an agent's decision trajectory, supporting auditing, accountability tracing, and further optimization.
