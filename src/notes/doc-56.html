Make Vibe Coding Safe: How to Test with Playwright

Amazon AGI Lab

✕

This talk explains how to use Playwright to make features built through "vibe coding" reliable and maintainable.

The speaker starts by introducing himself as a former core developer on Microsoft's Playwright, now doing browser automation at Amazon AGI Lab. He frames today's development pattern: people use coding agents like Cursor and Claude Code to "vibe code," quickly producing features that appear to work, but without safety guardrails, real user flows break easily — especially with repeated iteration afterward. The fix is still "write tests," except hand-writing traditional end-to-end tests (Selenium in the past, Playwright now) is too slow and painful.

He then contrasts two approaches. One lets the agent look only at source code, guess what buttons/text will appear on screen, and generate a Playwright test from that. That's fine for small projects, but once you have a large volume of node_modules, third-party packages, monorepo structure, and other complexity, the model gets easily overwhelmed by the code, guesses the wrong locator, and the test fails as soon as it runs, requiring repeated fixes.

The other approach wires the browser and Playwright MCP / Playwright Test MCP into the agent's loop. That way the agent doesn't just look at source code — it can actually open a browser, start a local server, click through the actual screen, and see the DOM and accessibility tree that React/Next.js finally renders. On every click, Playwright MCP feeds the whole accessibility tree (which buttons, labels, text exist) back to the model, letting it use real on-screen information to decide on the most stable locator and generate more reliable tests.

He demos this with a small Next.js "To-do / Console task" app: the goal is to write a test covering a two-step confirmation flow where a delete button requires clicking "delete" and then "sure." An agent that only looks at source code guesses at the UI structure, likely gets the selector wrong the first time, and needs several rounds of error-correction. An agent wired to Playwright Test MCP instead first uses MCP to confirm the server is running, then opens a browser, operates the actual page, clicks delete, and observes whether the list items and counter actually change, before writing a test based on this "real, observed on-screen behavior," and finally calling Playwright test to run it and confirm it passes.

In the middle section, he surveys the current tools that let an agent control a browser/tests: Playwright CLI, Playwright MCP, Playwright Test MCP (with test-runner capability), Vercel's Agent Browser, Google's Chrome DevTools MCP, and so on. The main point: if you're already using Playwright for end-to-end testing, the top recommendation is pairing it with Playwright Test MCP so the agent can directly run tests and manage the test list.

He also recaps the "MCP vs. Skills (CLI)" debate: before late 2025, MCP was token-hungry and inefficient, so people switched to CLI skills for a while to save cost. But Anthropic later improved MCP's execution and context usage, and his own small benchmark shows that MCP now actually consumes fewer context tokens than CLI — so the premise that "you must pick skills/CLI to save cost" no longer holds; in practice, just pick whichever form fits your overall workflow best.

His final practical advice: keep tests "small and sharp," not a pile of hard-to-maintain, bloated suites. For a vibe-coded app, he recommends writing one clear end-to-end spec per critical user journey, so the test suite stays small and green while still covering the flows that truly matter; run it on every commit, and always treat a failure as a bug to investigate, not something to tolerate as flakiness. Pair that with Playwright's screenshot, trace, and video artifacts, so both humans and agents can debug well after the fact. The overall message: building features has become very cheap, but "confidence" is still expensive — so make good use of Playwright plus MCP/agents to build a high-signal-to-noise end-to-end testing workflow, so that fast vibe coding can still ship safely.
